## 1. Introduction

### 1.1 Purpose
This document provides a detailed description of the Software Requirements Specification (SRS) for the Automated HVAC Diagnostic Tool. It is intended for stakeholders, including project managers, developers, testers, and end-users, to serve as a definitive guide for the system's functionality, constraints, and behavior. Any changes to the requirements must be formally amended in this document.

### 1.2 Project Scope
The Automated HVAC Diagnostic Tool is a software application designed to automatically detect, identify, and diagnose performance and operational faults within HVAC subsystemsâ€”specifically chillers and cooling towers. By leveraging both real-time sensed data and static fixed data, the tool aims to enhance energy efficiency, improve operational reliability, and reduce maintenance costs through proactive fault identification and actionable recommendations.

### 1.3 Definitions, Acronyms, and Abbreviations

| Term/Acronym | Definition |
| :--- | :--- |
| **HVAC** | Heating, Ventilation, and Air Conditioning |
| **MVP** | Minimum Viable Product |
| **SRS** | Software Requirements Specification |
| **Sensed Data** | Dynamic, real-time measurements collected from sensors (e.g., temperatures, pressures, flow rates). |
| **Fixed Data** | Static, unchanging information about the HVAC equipment (e.g., model numbers, design specifications, rated capacity). |
| **Fault** | A deviation from expected or normal operation in an HVAC subsystem that impacts performance or efficiency. |
| **Diagnostic Algorithm** | A predefined logical procedure or rule set used to analyze data and identify specific faults. |
| **Batch Processing** | The execution of diagnostic operations on a set of data provided via an input file, as opposed to real-time streaming. |

## 2. Overall Description

### 2.1 Product Perspective
This tool is a standalone, new software product. Its core function is to act as an analytical engine that sits between data sources (sensors, input files) and the end-user, providing diagnostic insights. The architecture must be modular to allow for the future inclusion of additional HVAC subsystems like boilers.

### 2.2 Product Functions
The primary functions of the system are:
1.  **Data Ingestion:** Acquire data from hardware sensors in real-time or from pre-formatted input files.
2.  **Fault Analysis:** Execute diagnostic algorithms on the ingested data to detect and identify specific faults in chillers and cooling towers.
3.  **Results Presentation:** Display clear and actionable diagnostic results to the user.
4.  **Data Management:** Securely store all input data and diagnostic results in a permanent database for historical analysis and reporting.

### 2.3 User Classes and Characteristics

| User Class | Characteristics | Key Responsibilities |
| :--- | :--- | :--- |
| **General User** | Technicians, facility managers. Has operational knowledge of HVAC systems. | View diagnostic results, run diagnostics on available data streams. |
| **Administrator** | System expert, IT staff. Has elevated system privileges. | Manage user accounts, configure system parameters (e.g., data source paths, algorithm thresholds), start/stop data acquisition services. |

### 2.4 Operating Environment
*   **Operating System:** The application shall be compatible with modern, supported Microsoft Windows operating systems (e.g., Windows 10, Windows 11, Windows Server 2019/2022).
*   **Software Dependencies:** A compatible relational database (e.g., Microsoft SQL Server, PostgreSQL) for permanent storage.

### 2.5 Design and Implementation Constraints
1.  **Platform Constraint:** The application shall be developed to run exclusively on Microsoft Windows-compatible operating systems.
2.  **Scope Constraint:** The initial release (MVP) shall provide diagnostic capabilities for chiller and cooling tower subsystems only. The system architecture must be designed to accommodate future expansion to other subsystems (e.g., boilers, AHUs) without major refactoring.
3.  **Security Constraint:** A role-based access control (RBAC) system must be implemented to restrict privileged functions (e.g., system configuration, user management) to users with the 'Administrator' role.

### 2.6 Assumptions and Dependencies
*   **Assumption:** Suitable hardware sensors are installed, calibrated, and accessible for real-time data acquisition.
*   **Assumption:** Input files for batch processing will be provided in a predefined, validated format (e.g., CSV, JSON).
*   **Dependency:** The availability and accuracy of diagnostic algorithms are dependent on domain expertise and may be updated in future versions.

## 3. System Features

This section details the high-level functional requirements of the system.

### 3.1 Fault Detection and Identification
**Description:** The system shall analyze input data to detect and identify predefined faults.
**Priority:** High

| Requirement ID | Requirement Description |
| :--- | :--- |
| **FR-010** | The system shall execute predefined diagnostic algorithms for chiller subsystems. |
| **FR-011** | The system shall execute predefined diagnostic algorithms for cooling tower subsystems. |
| **FR-012** | The diagnostic algorithms shall utilize both sensed (dynamic) data and fixed (static) equipment data. |
| **FR-013** | The system shall be capable of identifying a specific set of performance faults (e.g., reduced heat transfer efficiency, fouling). |
| **FR-014** | The system shall be capable of identifying a specific set of operational faults (e.g., sensor failure, pump failure). |

### 3.2 Diagnostic Results Display
**Description:** The system shall present diagnostic results in a clear and actionable manner to the user.
**Priority:** High

| Requirement ID | Requirement Description |
| :--- | :--- |
| **FR-020** | The system shall display a list of detected faults. |
| **FR-021** | For each detected fault, the system shall display a clear description of the fault. |
| **FR-022** | For each detected fault, the system shall display one or more potential root causes. |
| **FR-023** | For each detected fault, the system shall display one or more recommended corrective actions or fixes. |

### 3.3 Data Storage and Persistence
**Description:** The system shall persistently store all relevant data for future access and analysis.
**Priority:** High

| Requirement ID | Requirement Description |
| :--- | :--- |
| **FR-030** | The system shall store all raw sensed data with a timestamp and source identifier. |
| **FR-031** | The system shall store all fixed equipment data. |
| **FR-032** | The system shall store all diagnostic results, including fault IDs, timestamps, and severity levels. |
| **FR-033** | The system shall allow for the retrieval of historical data and results based on user-specified criteria (e.g., date range, equipment ID). |

### 3.4 Data Acquisition and Processing
**Description:** The system shall support multiple modes of data input.
**Priority:** High

| Requirement ID | Requirement Description |
| :--- | :--- |
| **FR-040** | The system shall support real-time data acquisition from connected hardware sensors (e.g., via Modbus, BACnet). |
| **FR-041** | The system shall support batch processing of data from formatted input files (e.g., CSV). |
| **FR-042** | The system shall validate the format and integrity of data from input files before processing. |

### 3.5 User and System Management
**Description:** The system shall manage user access and system configuration.
**Priority:** Medium

| Requirement ID | Requirement Description |
| :--- | :--- |
| **FR-050** | The system shall require user authentication via a login mechanism. |
| **FR-051** | The system shall enforce role-based permissions, granting 'General User' and 'Administrator' access levels. |
| **FR-052** | Only users with the 'Administrator' role shall be permitted to access system configuration menus. |
| **FR-053** | Only users with the 'Administrator' role shall be permitted to start, stop, or restart data acquisition services. |
| **FR-054** | Only users with the 'Administrator' role shall be permitted to manage (add, modify, deactivate) user accounts. |

## 4. External Interface Requirements

### 4.1 User Interfaces
The application shall feature a Graphical User Interface (GUI) with the following key components:
*   A **Login Screen** for user authentication.
*   A **Main Dashboard** showing system status and a high-level overview of active faults.
*   A **Diagnostics View** for displaying detailed fault information (description, causes, fixes).
*   A **Data Management View** for initiating batch processing and viewing historical data (accessible based on role).
*   An **Administration Panel** for user and system configuration (Administrator access only).

### 4.2 Hardware Interfaces
*   **HI-001:** The system shall interface with industry-standard HVAC sensors and controllers using protocols such as Modbus TCP/IP or BACnet IP.

### 4.3 Software Interfaces
*   **SI-001:** The system shall interface with a relational database management system (RDBMS) for all persistent data storage operations.

### 4.4 Communications Interfaces
*   **CI-001:** The system shall communicate over standard Ethernet TCP/IP networks to connect with hardware controllers and sensors.

## 5. Non-Functional Requirements

### 5.1 Performance Requirements
*   **PR-001:** The system shall process incoming real-time sensor data and execute relevant diagnostic algorithms with a latency of less than 5 seconds from data receipt to result generation.
*   **PR-002:** The user interface shall respond to user interactions (e.g., button clicks, page navigation) in less than 2 seconds.

### 5.2 Safety Requirements
*   The tool is for diagnostic and informational purposes only. It shall not directly control HVAC equipment (e.g., turn valves on/off, trip breakers). All recommended actions must be verified and executed by qualified personnel.

### 5.3 Security Requirements
*   **SR-001:** User passwords shall be stored using a strong, industry-standard hashing algorithm (e.g., bcrypt).
*   **SR-002:** The system shall log all user authentication attempts (success and failure).
*   **SR-003:** User sessions shall expire after a period of inactivity (e.g., 30 minutes).

### 5.4 Software Quality Attributes
*   **Maintainability:** The code shall be modular and well-documented to facilitate the future addition of diagnostic rules for new HVAC subsystems.
*   **Reliability:** The system shall achieve 99.5% uptime during standard operational hours. Data acquisition failures shall be logged and alerted.
*   **Usability:** The interface shall be intuitive for users with a technical HVAC background, avoiding overly complex computer science terminology.

### 5.5 Business Rules
*   Diagnostic algorithms are based on predefined physical models and rule sets provided by domain experts. These rules are considered intellectual property and are core to the application's functionality.
