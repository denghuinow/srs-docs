## 1. Introduction

### 1.1 Purpose
This document describes the functional and non-functional requirements for the High Assurance Transformation System (HATS) Graphical User Interface (GUI). The purpose of this software is to provide a platform-independent, user-friendly interface that simplifies interaction with the transformation-based software development tools of the HATS system. This SRS is intended for the project managers, developers, and testers involved in the implementation and verification of the HATS GUI.

### 1.2 Project Scope
The HATS GUI will be a standalone application that acts as a front-end to the HATS transformation engine (HATS-SML). It will allow users to manage transformation projects, visualize code structures as Syntax Derivation Trees (SDTs), execute transformation programs, and view the results. The core value is abstracting the command-line complexity of HATS-SML into an intuitive graphical environment, thereby increasing productivity and reducing the learning curve for new users.

## 2. Overall Description

### 2.1 Product Perspective
The HATS GUI is a new, self-contained component that interfaces with the existing HATS-SML backend engine. The relationship is that of a master (GUI) and slave (HATS-SML engine). The GUI will invoke the HATS-SML compiler and executor as external processes, parsing their outputs to present results to the user.

### 2.2 Product Functions
The high-level functions of the HATS GUI MVP are:
1.  **Application Management:** Create, delete, and manage HATS "applications," which are logical containers for transformation programs and target files.
2.  **Source Parsing and Visualization:** Parse transformation language programs and target files into SDTs and display them in a navigable tree view.
3.  **Transformation Execution:** Execute a selected transformation program on a selected target file using the HATS-SML engine.
4.  **Result Display and Navigation:** Display the resulting SDT and the pretty-printed text output of the transformation. Provide search and selection features within these views.

### 2.3 User Characteristics
The intended users are software developers and engineers with knowledge of formal methods and program transformation. They are technically proficient but require a more efficient workflow than a pure command-line interface offers.

### 2.4 Constraints
1.  **Implementation Language:** The application must be implemented in Java.
2.  **Platform Independence:** The compiled application must run identically, without code modification, on the following operating systems:
    *   Microsoft Windows 2000
    *   Sun Solaris
    *   Linux
3.  **Schedule:** All development and testing must be completed by December 2001.
4.  **User Interface Design:** The application must minimize the number of simultaneously open windows and dialog boxes to prevent user interface clutter and confusion.

### 2.5 Assumptions and Dependencies
*   **Critical Dependency:** The successful development of this GUI is dependent on the timely release and functional stability of HATS-SML version 2.0, which is assumed to be available by October 2001. Any delay in HATS-SML 2.0 will directly impact the project schedule.

## 3. Specific Requirements

### 3.1 External Interface Requirements

#### 3.1.1 User Interfaces
*   The main window shall be a Single Document Interface (SDI) or a well-managed Multiple Document Interface (MDI) to adhere to the constraint of minimizing open windows.
*   It shall contain a main menu bar with standard items (File, Edit, View, Tools, Help).
*   A project/application explorer pane shall be present for navigating files and associations.
*   A central multi-tabbed pane shall be used to display SDT views, pretty-printed text, and other content.
*   Context menus shall be available for relevant UI elements (e.g., files in the project explorer).

#### 3.1.2 Hardware Interfaces
*   The software shall not require any specific hardware other than a standard display, keyboard, and mouse.

#### 3.1.3 Software Interfaces
*   **HATS-SML Engine:** The GUI shall call the HATS-SML command-line executable (e.g., `hats-smlexec`) to perform parsing and transformation execution.
*   **Java Runtime Environment (JRE):** The application shall require a JRE version 1.3 or higher to be installed on the host system.

#### 3.1.4 Communications Interfaces
*   Not applicable for this version. All communication with HATS-SML is via local process execution.

### 3.2 Functional Requirements

#### 3.2.1 Application Management (FR-AM)

*   **FR-AM-1: Create Application**
    *   **Description:** The system shall allow the user to create a new HATS application.
    *   **Input:** User selects "New Application" from the File menu and provides a name and file system location.
    *   **Processing:** The system shall create a new directory structure and a configuration file at the specified location.
    *   **Output:** The new application is added to the application explorer pane.

*   **FR-AM-2: Modify File Associations**
    *   **Description:** The system shall allow the user to add transformation programs (.tl) and target files to an application and define the association between them.
    *   **Input:** User drags files into the application explorer or uses an "Add File" dialog.
    *   **Processing:** The system shall copy or link the files into the application's directory structure and update the internal application model.
    *   **Output:** The files are displayed within the application in the explorer pane.

*   **FR-AM-3: Delete/Remove Items**
    *   **Description:** The system shall allow the user to delete entire applications or remove individual files from an application.
    *   **Input:** User selects an application or file and chooses the "Delete" action.
    *   **Processing:** The system shall prompt for confirmation and then remove the selected item(s) from the file system and the internal model.
    *   **Output:** The item is removed from the application explorer pane.

#### 3.2.2 Parsing and SDT Visualization (FR-PV)

*   **FR-PV-1: Parse File**
    *   **Description:** The system shall parse a selected transformation program or target file into its Syntax Derivation Tree (SDT).
    *   **Input:** User double-clicks a file in the application explorer.
    *   **Processing:** The system shall invoke the HATS-SML parser for the file type and capture its output.
    *   **Output:** A new tab is opened in the central pane displaying a graphical, hierarchical tree view of the SDT.

*   **FR-PV-2: Navigate SDT**
    *   **Description:** The user shall be able to expand, collapse, and select nodes within the SDT view.
    *   **Input:** User clicks on tree node expanders/collapsers or selects a node.
    *   **Processing:** The view shall update to show/hide child nodes. Selection shall be highlighted.
    *   **Output:** The SDT view is updated accordingly.

*   **FR-PV-3: Search SDT**
    *   **Description:** The user shall be able to search for specific text or node types within the current SDT view.
    *   **Input:** User enters a search string in a search box and initiates the search.
    *   **Processing:** The system shall traverse the SDT and highlight all nodes matching the search criteria.
    *   **Output:** Matching nodes are highlighted, and the view may scroll to the first match.

#### 3.2.3 Transformation Execution (FR-TE)

*   **FR-TE-1: Execute Transformation**
    *   **Description:** The system shall execute a selected transformation program on a selected target file.
    *   **Input:** User selects a transformation program and a target file (e.g., via right-click context menu) and chooses "Execute."
    *   **Processing:** The system shall invoke the HATS-SML executor with the appropriate parameters.
    *   **Output:** The system displays a progress indicator and, upon completion, the results are displayed (see FR-RD).

#### 3.2.4 Result Display and Navigation (FR-RD)

*   **FR-RD-1: Display Result SDT**
    *   **Description:** Following a successful transformation, the system shall automatically parse and display the SDT of the transformed target file.
    *   **Input:** The output from the HATS-SML execution.
    *   **Processing:** The system shall parse the output into a new SDT model.
    *   **Output:** A new tab is opened in the central pane showing the SDT of the transformed code, adhering to the same navigation and search features as FR-PV.

*   **FR-RD-2: Display Pretty-Printed Text**
    *   **Description:** The system shall display a pretty-printed, textual representation of the transformation result.
    *   **Input:** The output from the HATS-SML execution.
    *   **Processing:** The system shall format the output text for readability.
    *   **Output:** A separate tab is opened in the central pane showing the formatted text. This view shall support standard text operations like selection, copying, and text-based search.

### 3.3 Non-Functional Requirements

#### 3.3.1 Performance Requirements
*   The GUI shall remain responsive during file parsing and transformation execution. Long-running operations shall be performed in background threads to prevent the UI from freezing.
*   Loading a medium-sized (â‰¤ 10,000 lines) SDT for visualization shall take no longer than 5 seconds on standard hardware for the year 2001.

#### 3.3.2 Safety Requirements
*   The application shall not modify any source files without explicit user instruction (e.g., executing a transformation). All file deletions shall require user confirmation.

#### 3.3.3 Security Requirements
*   The application is not designed for multi-user environments and has no specific security requirements beyond the native file system permissions of the host OS.

#### 3.3.4 Software Quality Attributes

*   **Availability:** Not a critical requirement for this desktop application.
*   **Maintainability:** The Java code shall be well-documented and structured to facilitate future enhancements.
*   **Portability:** As a core constraint, the application must be fully platform-independent across the specified operating systems (Windows 2000, Sun Solaris, Linux). This shall be verified by testing on all three platforms.
*   **Usability:**
    *   The user interface shall be intuitive for developers familiar with IDE-style applications.
    *   The constraint to minimize open windows shall be strictly enforced to reduce user cognitive load.
    *   Common actions (open, execute, save) shall have keyboard shortcuts.
